ARM 32-bit Assembly — rpm_to_duty_asm()

The RPM-to-duty-cycle conversion runs in hand-written ARMv7 inline assembly using sdiv, mul, and movlt/movgt for inline clamping. Input RPM is clamped to 400–5700 RPM and mapped linearly to 4–100% duty cycle. The app dial sends up to 7000 RPM, but values above 5700 saturate at 100% duty.

bt_motor_server.c — rpm_to_duty_asm()
int min_rpm  = 400,  max_rpm  = 5700;  /* RPM input clamp range     */
int min_duty = 4,    max_duty = 100;   /* duty cycle % output range */
int span_duty = 96,  span_rpm  = 5300; /* (max − min) spans         */

__asm__ volatile (
    "cmp   %[val], %[minR]  \n"   /* clamp low                 */
    "movlt %[val], %[minR]  \n"
    "cmp   %[val], %[maxR]  \n"   /* clamp high                */
    "movgt %[val], %[maxR]  \n"
    "sub   %[val], %[val], %[minR]  \n"  /* shift to 0-base           */
    "mul   %[val], %[val], %[spanD] \n"  /* scale by duty span        */
    "sdiv  %[val], %[val], %[spanR] \n"  /* divide by RPM span        */
    "add   %[val], %[val], %[minD]  \n"  /* add minimum duty offset   */
    : [val]   "+r"(rpm)
    : [minR]  "r"(400),  [maxR]  "r"(5700),
      [minD]  "r"(4),    [maxD]  "r"(100),
      [spanD] "r"(96),   [spanR] "r"(5300)
    : "cc"
);
Communication

Packet Protocol

Phone-to-Pi uses 4-byte binary packets, sent on every user state change. Pi-to-Phone uses newline-terminated ASCII decimal strings sent every 500 ms, read by Android's BufferedReader.readLine(). The Pi only sends when a client socket is open (client_fd ≥ 0).

MainActivity.kt — sendCommand()
fun sendCommand(isPowerOn: Boolean, direction: String, rpm: Float) {
    val targetRpm = rpm.toInt()
    val payload   = ByteArray(4)
    payload[0] = if (isPowerOn)       1 else 0
    payload[1] = if (direction == "CCW") 1 else 0
    payload[2] = ((targetRpm shr 8) and 0xFF).toByte()  // high byte
    payload[3] = (targetRpm and 0xFF).toByte()           // low byte
    outputStream?.write(payload)
}

RPM Measurement — 3-blade propeller, rising-edge detection

The rpm_worker thread polls GPIO 25 every 500 µs counting rising edges. With PULSES_PER_REV = 3.0 (3 blades per revolution), RPM is computed over a 500 ms sample window:

RPM = (60.0 × pulse_delta) / (3.0 × elapsed_seconds)

The computed value is stored under mutex and transmitted to the phone in the same 500 ms pass.

🔁

L293D Direction Logic

CW (DIR_FORWARD): IN2 = 0, IN1 = PWM, LED = PWM
CCW (DIR_REVERSE): IN1 = 0, IN2 = PWM, LED = PWM
STOP / OFF: IN1 = 0, IN2 = 0, LED = 0

PWM is generated by a dedicated POSIX thread. Period = 2000 µs (500 Hz). On-time = (2000 × duty%) / 100 µs. Timing via nanosleep().

Capabilities

What's Implemented

Everything below is working in the current CP3 codebase. Automatic closed-loop modes are planned for the final checkpoint.

🎛️

Circular Dial — 400 to 7000 RPM

Touch drag maps angle (135°–405°) to RPM. Sends a 4-byte binary command on every gesture update. Starting value is 400 RPM. Duty cycle saturates at 100% above 5700 RPM.

✓ CP3
🟢

On / Off Power Toggle

Center button toggles power. OFF sends power=0, which forces DIR_STOP and duty=0 on the Pi. Button shows green (ON) or red (OFF). State included in every 4-byte packet.

✓ CP3
🔄

CW / CCW Direction

Two circular side buttons select rotation direction. The command is only re-transmitted when direction actually changes. CW = IN1 active, CCW = IN2 active on the L293D.

✓ CP3
📡

Live RPM Feedback

IR sensor rising edges counted over a 500 ms window on the Pi. Actual RPM sent as ASCII string every 500 ms. Android reads it on a dedicated IO coroutine and updates the display in real time.

✓ CP3
⚡

Software PWM @ 500 Hz

Dedicated POSIX thread generates a 2000 µs period PWM signal via nanosleep(). On-time = (period × duty%) / 100. No hardware PWM dependency.

✓ CP3
🔵

Bluetooth RFCOMM Reconnect

RFCOMM socket uses BT_SECURITY_LOW, bypassing the Pi's GUI pairing prompt. Pi re-runs bluetoothctl discoverable on after every disconnect. Phone can reconnect after "Forget Device" without touching the Pi.

✓ CP3
🔒

MAC Address Allowlist

Pi compares the MAC of every incoming connection against a hardcoded constant. Any non-matching address is rejected with an immediate close(). Only the one authorized phone can drive the motor.

✓ CP3
🤖

Automatic Maintain Mode

Closed-loop PID control using actual vs. target RPM from the IR sensor. Pi adjusts duty cycle automatically to hold the setpoint. Planned for the final checkpoint.

CP4 Final
🔗

Automatic Match Mode

Second IR sensor monitors an external propeller. The system dynamically matches its detected RPM. Requires second sensor and feedback loop. Planned for final checkpoint.
